Naming Conventions in C

Back to Main Page | Best Practices Overview

General Principles

Good naming is crucial for code readability and maintainability. Follow these general principles:

Naming Principles

  • Clarity: Names should clearly indicate the purpose or meaning
  • Consistency: Use consistent naming patterns throughout your code
  • Descriptiveness: Names should be descriptive but not excessively long
  • Pronounceability: Names should be easy to pronounce and discuss

When in doubt, prioritize clarity over brevity. A slightly longer but clear name is better than a short but cryptic one.

Variable Naming

Variables should have meaningful names that indicate their purpose and content.

Common Variable Naming Conventions

C programmers typically use one of these conventions:

Snake Case

Words are separated by underscores, all lowercase:

int user_age;
float account_balance;
char first_name[50];

Camel Case

First word starts with lowercase, subsequent words are capitalized:

int userAge;
float accountBalance;
char firstName[50];

Choose one convention and use it consistently throughout your project. Snake case is more common in traditional C code, while camel case is often seen in projects influenced by C++ or Java conventions.

Good vs. Bad Variable Names

Good Variable Names

int age;                  // Simple and clear
float temperature_celsius;  // Specific with units
char user_input[100];      // Describes purpose
bool is_valid;             // Boolean with 'is_' prefix
int num_students;          // Count with 'num_' prefix

Bad Variable Names

int a;                    // Too short and vague (except for loop counters)
float temp;                // Ambiguous (temperature? temporary?)
char str[100];            // Too generic
int flag;                 // Unclear purpose
float x1, x2, x3;         // Meaningless sequence

Avoid using single-letter variable names except for loop counters (i, j, k) or mathematical conventions (x, y, z for coordinates).

Function Naming

Function names should clearly describe what the function does.

Verb-Noun Pattern

Functions typically perform actions, so names often start with a verb:

void calculate_average(int* values, int count);
int find_maximum(int* array, int size);
bool validate_input(const char* input);
void sort_array(int* array, int size);
char* get_user_name();

Boolean Functions

Functions that return boolean values should be named to make conditional statements readable:

bool is_prime(int number);
bool has_permission(const char* user, const char* resource);
bool can_proceed(int status);

// Usage in conditionals
if (is_prime(number)) {
    // Do something
}

Common prefixes for boolean functions include: is_, has_, can_, should_, and contains_.

Good vs. Bad Function Names

Good Function Names

void initialize_game();       // Clear purpose
int count_words(const char* text);  // Describes action and return value
void print_error_message(int code); // Specific action
bool is_valid_email(const char* email); // Clear boolean function

Bad Function Names

void process();               // Too vague
int do_it(int x);              // Unclear purpose
void handle_stuff(void* data);   // Non-specific
bool check(int value);          // What is being checked?

Constant Naming

Constants should be named to clearly distinguish them from variables.

Macro Constants

Macro constants defined with #define are typically written in uppercase with underscores:

#define MAX_BUFFER_SIZE 1024
#define PI 3.14159
#define ERROR_FILE_NOT_FOUND 404
#define DEFAULT_TIMEOUT_MS 5000

Const Variables

For const variables, you can use either all uppercase or the regular variable naming convention:

/* All uppercase (like macros) */
const int MAX_USERS = 100;
const float GRAVITY = 9.81;

/* Regular variable naming convention */
const int max_users = 100;
const float gravity = 9.81;

Be consistent with your chosen convention for constants throughout your codebase.

Type Naming

Custom types should have clear, descriptive names.

Struct and Union Names

Struct and union names often use PascalCase (first letter of each word capitalized) or snake_case:

/* PascalCase */
struct Person {
    char name[50];
    int age;
};

/* snake_case */
struct user_account {
    char username[20];
    char password[64];
};

Typedef Names

When using typedef, the new type name often ends with _t or uses PascalCase:

/* With _t suffix */
typedef struct {
    int x;
    int y;
} point_t;

/* PascalCase */
typedef struct {
    float real;
    float imag;
} Complex;

Enum Names

Enum types typically follow the same convention as structs, while enum values often use uppercase like constants:

enum Color {
    RED,
    GREEN,
    BLUE
};

typedef enum {
    SUCCESS,
    ERROR_INVALID_INPUT,
    ERROR_OUT_OF_MEMORY,
    ERROR_FILE_NOT_FOUND
} Status;

Some C projects prefix enum values with the enum name or an abbreviation to avoid naming conflicts: COLOR_RED, COLOR_GREEN, etc.

File Naming

File names should reflect their contents and follow a consistent pattern.

Source and Header Files

File names typically use lowercase with underscores or hyphens:

Common File Naming Patterns

  • string_utils.c / string_utils.h
  • file-io.c / file-io.h
  • user_management.c / user_management.h

Module Organization

For larger projects, group related files with consistent prefixes:

ui_button.c
ui_menu.c
ui_dialog.c

db_connect.c
db_query.c
db_transaction.c

net_socket.c
net_http.c
net_protocol.c

Keep file names short but descriptive. Avoid generic names like utils.c or helpers.c unless they truly contain miscellaneous utilities.

Consistency is Key

The most important aspect of naming conventions is consistency.

Benefits of Consistent Naming

  • Improves code readability
  • Makes code more predictable
  • Reduces cognitive load when reading code
  • Makes it easier for new developers to understand the codebase
  • Helps identify the purpose and type of identifiers at a glance

If you're working on an existing project, follow the conventions already established, even if they differ from your personal preferences. For new projects, document your naming conventions to ensure consistency among team members.

Mixing different naming conventions within the same codebase can lead to confusion and errors. Choose one style and stick with it.
